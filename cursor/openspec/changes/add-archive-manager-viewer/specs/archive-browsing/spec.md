## 新增需求

### 需求:免解压列出压缩包内容

系统必须在打开压缩包时仅读取其目录信息以列出内部条目，禁止为列表展示而将整个压缩包解压到磁盘。条目至少包含名称、大小、是否为目录及推断的文件类型。

#### 场景:打开 zip 秒速列出条目
- **当** 用户打开一个 zip/cbz 压缩包
- **那么** 系统不解压即列出包内所有文件与文件夹

#### 场景:进入子目录浏览
- **当** 用户点击包内的一个文件夹
- **那么** 系统展示该文件夹下的条目，并支持返回上一级

### 需求:统一格式抽象与分级支持

系统必须通过统一的压缩包访问抽象（Provider）接入不同格式，使上层浏览与查看逻辑不依赖具体格式。每种格式必须声明其是否支持随机访问（直接抽取单个条目）。

#### 场景:声明随机访问能力
- **当** 系统加载某格式的压缩包
- **那么** 该格式的 Provider 必须报告其是否支持随机访问

#### 场景:首版支持 zip/cbz
- **当** 用户打开 zip 或 cbz 压缩包
- **那么** 系统必须能列出条目并对单个条目开流（满级免解压体验）

### 需求:不支持随机访问格式的降级处理

对于不支持随机访问的格式（如 7z solid、tar.gz），系统必须采用后台解压到应用私有缓存的降级策略，并在准备期间向用户给出明确提示，禁止在界面上谎称已"秒开"。

#### 场景:打开重型格式时显示准备状态
- **当** 用户打开一个不支持随机访问的压缩包
- **那么** 系统在后台解压到缓存的同时显示"正在准备…"提示，准备完成后再展示内容

#### 场景:缓存超限时清理
- **当** 降级解压产生的缓存超过设定上限
- **那么** 系统按 LRU 策略清理旧缓存

### 需求:浏览嵌套压缩包

系统必须支持进入压缩包内部的嵌套压缩包继续浏览。

#### 场景:打开包内的压缩包
- **当** 用户点击压缩包内的另一个压缩包条目
- **那么** 系统将其作为压缩包打开并展示其内部条目
